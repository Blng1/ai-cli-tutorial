Skills 体系

Skills 与 Subagents — AI CLI 工具的两层可扩展能力体系。学会封装和复用，让 AI 工具适配你的工作流。

01 两层能力体系

AI CLI 工具的可扩展能力分为两个核心层次：Skills 负责能力封装与知识复用，Subagents 负责隔离执行与任务委派。两者协同工作，构成完整的能力扩展框架。

Commands 已合并入 Skills

在 Claude Code 中，自定义 Slash Commands 已与 Skills 合并。.claude/commands/review.md 和 .claude/skills/review/SKILL.md 都会创建 /review 命令，效果完全相同。旧的 .claude/commands/ 文件仍然有效，但 Skills 提供了更多高级功能：支持辅助文件目录、控制谁可以触发、以及 AI 自动加载。建议新项目直接使用 Skills 格式。

02 Skills 深度解析 KEY

Skills 是 AI CLI 工具最核心的可扩展机制。它不只是一段 Prompt，而是一个包含知识、规则、模板和工具配置的完整能力单元。掌握 Skills 是从"使用 AI"到"定制 AI"的关键跃迁。

什么是 Skill

CONCEPT

一个 Skill 是一个结构化的能力描述文件，包含名称、触发条件、执行逻辑和工具约束。它本质上是一个"可复用的 Prompt 模板 + 工具配置 + 辅助资源"的组合。

Skill 具有以下核心特性：

  • 双向触发 — 用户通过 /skill-name 手动调用，或 AI 根据 description 自动匹配激活
  • 辅助资源 — Skill 目录可包含模板、示例代码、脚本等辅助文件，按需加载，不浪费上下文
  • 工具约束 — 通过 allowed-tools 限制 Skill 激活时可用的工具，如只读模式
  • 调用控制 — 精细控制谁能触发：仅用户、仅 AI、或两者都可
  • 子代理执行 — 通过 context: fork 在独立子代理中运行，不消耗主会话上下文
  • 跨会话持久 — Skill 文件存储在项目或用户目录中，不会随会话结束而消失

Skill 文件结构与配置

STRUCTURE

Skill 文件使用 Markdown 格式，存放在 .claude/skills/<skill-name>/ 目录下，主文件名为 SKILL.md。文件由 YAML frontmatter (配置) 和 Markdown 正文 (指令) 两部分组成。Claude Code Skills 遵循 Agent Skills 开放标准，并在此基础上扩展了调用控制、子代理执行等高级特性。

.claude/skills/fix-issue/SKILL.md
--- # ── Agent Skills 标准字段 ── name: fix-issue description: > Fix a GitHub issue by number. Read the issue, understand requirements, implement the fix, write tests, and create a commit. license: MIT metadata: author: my-team version: "1.2" # ── Claude Code 扩展字段 ── disable-model-invocation: true argument-hint: [issue-number] allowed-tools: Read Grep Glob Bash(gh *) --- Fix GitHub issue $ARGUMENTS following our standards. 1. Read the issue: !`gh issue view $ARGUMENTS` 2. Understand the requirements 3. Implement the fix 4. Write tests 5. Create a commit

标准目录结构 — 遵循 Agent Skills 开放规范，支持三类可选子目录：

Skill 目录结构 (Agent Skills Standard)
.claude/skills/ ├── fix-issue/ │ └── SKILL.md # 主文件 (必须) ├── commit/ │ └── SKILL.md └── deep-research/ ├── SKILL.md # 主文件：概述和导航 (<500 行) ├── scripts/ # 可执行代码 (Python/Bash/JS) │ └── extract.py ├── references/ # 参考文档 (按需加载) │ ├── REFERENCE.md │ └── FORMS.md └── assets/ # 静态资源 (模板/图表/数据) └── template.json

渐进式上下文加载 (Progressive Disclosure) — Skill 内容分三层按需注入，避免浪费上下文窗口：

层级 加载时机 Token 开销 内容
L1 元数据 启动时 (所有 Skill) ~100 tokens/Skill frontmatter 的 name + description
L2 指令 激活时 <5,000 tokens (建议) SKILL.md 正文完整内容
L3+ 资源 按需加载 无上限 scripts/ references/ assets/ 目录文件

Frontmatter 字段参考 — 分为行业标准字段与 Claude Code 扩展字段两组：

Agent Skills 开放标准字段 (agentskills.io)

字段 必填 约束 说明
name 是 1-64 字符，仅小写字母 + 数字 + 连字符，不能以 - 开头/结尾，不能含 --，须与目录名一致 Skill 唯一标识符，也是 /slash 命令名
description 是 1-1024 字符，应包含触发关键词和使用场景 AI 自动匹配的核心依据，描述 Skill 做什么以及何时使用
license 否 许可证名称或引用打包的许可证文件 如 MIT、Apache-2.0
compatibility 否 最长 500 字符 环境要求：目标工具、系统包、网络访问等。如 Requires git, docker, jq
metadata 否 任意 key-value 映射 (字符串值) 自定义元数据。如 author、version、tags
allowed-tools 否 空格分隔的工具名列表 EXPERIMENTAL 预批准的工具列表。如 Read Grep Glob

Claude Code 扩展字段 (code.claude.com)

字段 默认值 说明 典型场景
disable-model-invocation false 设为 true 时 AI 不会自动触发，仅用户手动 /name 调用。description 也不会加载到上下文 /deploy, /commit 等有副作用的操作
user-invocable true 设为 false 时从 / 菜单隐藏，仅 AI 根据 description 自动加载 api-conventions 等背景知识
argument-hint — 自动补全时显示的参数提示 [issue-number]、[filename] [format]
context — 设为 fork 时在独立子代理中执行，隔离上下文 deep-research 等高产出任务
agent general-purpose 当 context: fork 时，指定使用的子代理类型 (内置或自定义) Explore、Plan、自定义 agent
model — 指定执行 Skill 时使用的模型 sonnet / opus / haiku
hooks — 限定作用于此 Skill 生命周期的钩子函数 Skill 激活/完成时触发自定义脚本

调用控制矩阵 — 两个字段组合决定 Skill 的可见性和触发方式：

配置 用户可调用 AI 可调用 上下文加载
(默认) 是 是 description 始终在上下文，正文按需加载
disable-model-invocation: true 是 否 description 不在上下文，用户调用时才加载正文
user-invocable: false 否 是 description 始终在上下文，AI 激活时加载正文

参数替换与动态上下文注入

ADVANCED

Skill 正文支持字符串替换变量和 Shell 预处理命令，让 Skill 能够接收参数并动态注入运行时数据。

字符串替换变量

变量 说明 示例
$ARGUMENTS 调用 Skill 时传入的所有参数。若正文中未包含该变量，参数自动追加到末尾 /fix-issue 423 → "423"
$ARGUMENTS[N] / $N 按 0-based 索引访问单个参数。$0 是 $ARGUMENTS[0] 的简写 /migrate $0=$1→$2
${CLAUDE_SESSION_ID} 当前会话 ID，适用于日志记录或创建会话特定文件 logs/${CLAUDE_SESSION_ID}.log

动态上下文注入 — !`command` 语法在 Skill 内容发送给 AI 之前执行 Shell 命令，将命令输出替换占位符：

!`command` 预处理示例
--- name: pr-summary description: Summarize changes in a pull request context: fork agent: Explore allowed-tools: Bash(gh *) --- ## Pull request context - PR diff: !`gh pr diff` # 执行 gh CLI，输出替换此处 - PR comments: !`gh pr view --comments` - Changed files: !`gh pr diff --name-only` ## Your task Summarize this pull request...

执行顺序：!`command` 先运行 → 输出替换占位符 → 渲染后的完整内容发送给 AI。这是预处理，AI 只看到最终结果。

Skill 的存储与触发流程

FLOW

理解 Skill 的存储层级和触发流程有助于编写更精准的配置。

1. 存储位置与作用范围

存放位置决定谁可以使用该 Skill，同名时高优先级覆盖低优先级：

级别 路径 适用范围 优先级
Enterprise Managed Settings 配置 组织内所有用户 最高
个人级 ~/.claude/skills/<name>/SKILL.md 该用户所有项目 ↓
项目级 .claude/skills/<name>/SKILL.md 仅当前项目 (可提交 Git) ↓
Plugin <plugin>/skills/<name>/SKILL.md 插件启用的项目 (命名空间隔离) 不冲突

Monorepo 自动发现 — 编辑子目录文件时，Claude Code 会自动发现嵌套的 .claude/skills/ 目录。例如编辑 packages/frontend/ 中的文件时，也会加载 packages/frontend/.claude/skills/ 中的 Skill。

2. 渐进式加载流程 (Progressive Disclosure)

会话启动时，AI 仅加载所有 Skill 的 name + description (~100 tokens/Skill)。当用户请求匹配某 Skill 的描述时，AI 按需加载完整 SKILL.md 正文。scripts/ 等资源文件仅在 Skill 指令引用时才读取。

3. 触发方式

  • 用户手动触发 — 输入 /skill-name args，args 替换 $ARGUMENTS，argument-hint 在自动补全时提供参数格式提示
  • AI 自动触发 — 用户请求匹配 description 关键词时，AI 自动激活对应 Skill
  • 禁止自动触发 — 设置 disable-model-invocation: true，仅限用户手动调用
触发流程示意
# 场景 1：用户手动触发 (带参数) 用户输入: /fix-issue 423 → AI 加载 fix-issue/SKILL.md 完整内容 → $ARGUMENTS 被替换为 "423" → !`command` 预处理执行 → AI 按指令执行修复流程 # 场景 2：AI 自动匹配 (渐进式加载) 用户输入: "帮我解释一下这段代码怎么工作的" → AI 匹配 explain-code 的 description (L1: ~100 tokens) → 自动加载 Skill 正文 (L2: <5000 tokens) → 按需读取 references/ 等文件 (L3: 按引用) → 按 Skill 规则输出 (类比 + 图表 + 步骤) # 场景 3：fork 到子代理执行 用户输入: /deep-research auth module → 创建独立子代理上下文 (context: fork) → agent 字段决定执行环境 (Explore / Plan / 自定义) → Skill 正文作为子代理任务 → 结果摘要返回主会话 # 场景 4：索引参数访问 用户输入: /migrate-component SearchBar React Vue → $0=SearchBar $1=React $2=Vue

实用 Skill 示例

EXAMPLES

基于社区使用数据和开发者真实工作流筛选，每个示例展示不同的 Skill 特性

GIT commit

智能 Git 提交。分析 diff 内容，自动生成符合 Conventional Commits 规范的提交信息。设置 disable-model-invocation: true 确保仅用户手动触发，防止 AI 擅自提交。

REVIEW review-pr

PR 代码审查。通过 allowed-tools: Read, Grep, Glob 限制为只读模式，按严重程度分级反馈：Critical / Warning / Suggestion。

LEARN explain-code

代码解释。AI 根据 description 自动匹配"这段代码怎么工作"类提问。输出包含日常类比、ASCII 流程图和分步解析，是官方推荐的经典 Skill 示例。

RESEARCH deep-research

深度代码研究。设置 context: fork + agent: Explore，在只读子代理中运行，不消耗主会话上下文，适合大范围代码库探索。

TEST test-driven-development

测试驱动开发。社区下载量 Top 级 Skill，引导 AI 严格遵循 RED-GREEN-REFACTOR 循环：先写失败测试，再写最小实现，最后重构。

META skill-creator

Skill 创建向导。官方内置的元技能，引导你编写规范的 SKILL.md — 从 description 措辞、frontmatter 配置到辅助文件组织，确保产出的 Skill 符合最佳实践。

MULTI-AGENT omo

Oh-My-OpenCode 多代理编排。社区 2.1K Stars，内置 7 个专业代理协同工作 — Sisyphus (编排)、Oracle (架构)、Librarian (文档)、Explore (搜索) 等，按任务类型和风险自动选择最小代理组合，适合复杂开发任务。

03 Agent Skills 开放标准

Skills 不只是某个工具的专属特性。Anthropic 将 Skills 作为开放标准 (Agent Skills) 发布，多个 AI 工具已采纳该标准。学一次，到处用。

开放标准与生态

OPEN STANDARD

Agent Skills 是由 Anthropic 发起并于 2025 年 12 月 18 日发布的开放标准 (agentskills.io)，定义了 AI 工具能力扩展的通用规范。核心理念是：同一个 Skill 文件应当能跨工具使用，无需修改。

  • 标准规范 — 统一的 SKILL.md 格式 (YAML frontmatter + Markdown body)，标准字段 (name / description / license / compatibility / metadata / allowed-tools)，标准目录结构 (scripts/ / references/ / assets/)
  • 跨工具兼容 — Claude Code (发起者)、OpenAI Codex (API)、OpenCode、Gemini CLI、GitHub Copilot CLI、Kilo Code、Cursor 等均已支持或正在接入
  • 社区生态 — 社区市场已索引超过 185,000 个公开 Skills，涵盖前端设计、测试、部署、文档生成等各个领域
  • 分发机制 — 通过插件系统或直接目录分享，团队可在 Git 仓库中共享 Skills
  • 渐进式加载 — 三层上下文管理 (元数据 ~100 tokens → 指令 <5000 tokens → 资源按需)，优化上下文窗口利用率
  • 验证工具 — 官方提供 skills-ref 参考库，skills-ref validate ./my-skill 一键检查 frontmatter 合规性

类比理解：Agent Skills 之于 AI CLI 工具，就像 npm 包之于 Node.js — 一个可复用、可分享、跨平台的能力单元。各工具在标准基础上自行扩展特有字段 (如 Claude Code 的 context: fork、hooks)，但标准字段保持互通。

跨工具兼容性说明

OpenAI Codex 通过 POST /v1/skills API 上传 Skill (zip 包或多文件)，frontmatter 的 name 和 description 用于 Skill 发现和路由，与 Agent Skills 标准完全一致。OpenCode 同时扫描 .opencode/skills/、.claude/skills/ 和 .agents/skills/ 三个路径，最大化兼容性。Cursor 使用 .cursor/rules/ 中的 .mdc Rule 文件，按 glob 模式匹配而非按描述自动触发，与 Agent Skills 标准有差异，但正在逐步增加兼容性。

05 Subagents 概述

Subagent 是能力体系的第二层，能够在独立上下文中自主完成多步骤任务。本节做简要介绍，详细内容请参阅第 05 章。

Subagent 的核心特征

AGENT

Subagent 是在独立上下文窗口中运行的专用子进程。它不与主会话共享 token 空间，每个 Subagent 拥有自己的系统提示词、工具权限和执行环境。

  • 自主规划 — 将复杂任务分解为多个子步骤，按序执行
  • 工具调用 — 读取文件、执行命令、编辑代码、搜索信息
  • Skill 预加载 — 通过 skills 字段在启动时注入 Skill 知识
  • 上下文隔离 — 独立上下文窗口，执行完毕后仅返回摘要给主会话
  • 模型选择 — 可指定使用更快的 Haiku 或更强的 Opus 模型

Subagents 分为两类：

  • Subagents (子代理) — 在单个会话内委派任务，定义在 .claude/agents/ 目录下。内置的有 Explore (只读搜索)、Plan (规划) 和 general-purpose (通用)
  • Agent Teams (代理团队) — 多个代理跨会话协作，各自拥有完全独立的上下文，通过消息和任务列表协调工作，适合大型并行任务

Skills 是 AI 工具个性化的核心

当你发现自己反复给 AI 相同的指令时，就应该把它封装成一个 Skill。好的 Skills 系统能让你的 AI 工具从"通用助手"变成"定制专家"。建议从团队最常见的操作开始，逐步构建 Skill 库。一个成熟团队的 Skill 库通常包含 10-20 个核心 Skill，覆盖从代码生成、测试、审查到部署的完整开发流程。得益于 Agent Skills 开放标准，这些 Skill 可以在 Claude Code、OpenCode、Gemini CLI 等多个工具间直接复用。
